# Example: Feature Change

## Prompt

```text
before you build: A user asked us to add team workspaces and roles. Should we build it now?
```

## What The Skill Should Focus On

- Whether this is one user request or repeated target-user behavior.
- Whether the missing feature blocks payment, retention, trust, or core usage.
- Whether the product has proven individual usage first.
- Whether team features hide a weaker onboarding, pricing, or positioning problem.

## Good Output Shape

```markdown
## Feature Reality Check

Requested change:
- Add team workspaces and roles.

Biggest risk:
- You may be building team-plan complexity before one user has proven repeated value.

What may be wrong:
- This looks like a user request trap unless multiple paying users cannot adopt the product without roles.

Validate first:
1. Ask the user what breaks today without team workspaces.
2. Check whether at least 3 target accounts refused payment because this was missing.
3. Offer a manual workaround for one account before building a general permissions system.

Recommendation:
Validate first
```
